Big Picture: How a Migration Session Works
From picking a screen to shipping OTA — every step a developer (or Claude agent) follows.
Phase A

Ground Truth

Playwright crawls every screen in 4 user states. Captures HAR, screenshots, DOM, WebSocket frames. Generates SCREEN_SPEC.md.

Phase B

7-Gate Deep Migration

Each screen passes 7 sequential gates: Observe, Data Map, Design, Build, Analyze, Re-Verify, Score. Evidence required to advance.

Phase C

Continuous Verification

Regression pool, L1-L5 tests, CI pipeline, device testing, Shorebird OTA patches. Every done screen stays verified.

End-to-End Flow per screen
1

Single source of truth. Find the first screen with an incomplete gate. No asking — the file tells you what to do next.

AUTO-ROUTED

OUTPUT: DESIGN.md

Session Rules

Can Batch

  • Gates 1+2 for same screen (both read-only)
  • Gate 1 for multiple screens
  • Gate 2 for multiple screens

One Per Session

  • Gate 4 (build) — produces code
  • Gate 5 (analyze) — requires execution
  • Gate 6 (re-verify) — requires device
  • Gate 7 (score) — decision gate
Gate Details
Click any gate to expand full instructions and required outputs.
1
OBSERVE
Read-only
  • Read ground-truth/{screen}/SCREEN_SPEC.md
  • Read HAR files — note every API endpoint, header, response shape
  • Read WebSocket frames — note subscriptions, message formats
  • Read screenshots across all 4 user states
  • If existing_code: true — note what Flutter covers vs what it misses
Output
Understanding only. Update state.yaml: gates.observe = done
2
TRACE DATA
Read-only
  • Follow chain: HAR endpoint → web hook/component → GraphQL resolver → REST → response schema
  • Document global store (Redux/Jotai) state shape
  • All possible states: loading, null, empty array, error, stale
  • If existing_code: true — note which endpoints Flutter calls, which headers it sends
Output
ground-truth/{screen}/DATA_MAP.md
3
DESIGN
Write
  • Define Flutter state model: providers, possible states, UI per state
  • If existing_code: compare existing providers, note refactoring needed
  • Check ADRs — update if new pattern discovered
Output
ground-truth/{screen}/DESIGN.md + ADR update if needed
3.5
SECTION INVENTORY
Hard Gate
  • Read the web source component for this screen
  • List EVERY section in render order: container CSS, decorative layers, sub-component count, click handlers, data source, conditional rendering
  • Cross-reference against existing Flutter code — mark YES/NO for each section
  • This gate BLOCKS Gate 4 — cannot write code without it
Output
ground-truth/{screen}/SECTION_INVENTORY.md
4
BUILD
Code + Device
  • Build ONE section at a time from SECTION_INVENTORY.md
  • Extract Tailwind classes → exact Dart hex colors
  • Note decorative layers (gradients, glows, overlays)
  • Note ALL click handlers and destinations
  • Write Flutter code → flutter analyze → deploy → screenshot
  • Verifier sub-agent compares web vs Flutter screenshot
  • PASS (≥95%) → lock section. FAIL → fix, retry (max 5)
  • After all sections locked: full-screen verify + write tests (min 4 state tests)
Output
Flutter source files + .migration/verify/{screen}/flutter_iter_N.png
5
ANALYZE
Verify
  • flutter analyze — must be 0 issues
  • flutter test — all pass
  • Must have 4+ state-specific tests: loading, empty, populated, error
Output
Test results logged in state.yaml evidence
6
RE-VERIFY
6-Dimension
  • 6a Network: Diff Flutter HAR vs web HAR — same endpoints, headers, payloads
  • 6b Visual: Screenshot comparison (golden files or device screencap)
  • 6c State: All 4 user states tested (from Gate 5)
  • 6d Interaction: Install APK, walk user flows, verify nav/forms/realtime
  • 6e Analytics: Verify events from SCREEN_SPEC called in Flutter code
  • 6f Performance: Frame rendering, screen transition latency, 60fps scroll
Output
ground-truth/{screen}/PARITY_REPORT.md with G1-G6 scores
7
SCORE & VERDICT
Decision
  • Check G1-G6 against thresholds: G1-G4 ≥ 95%, G5 ≥ 90%, G6 ≥ 80%
  • Composite must be ≥ 95% (weighted: G1:20% G2:15% G3:25% G4:20% G5:10% G6:10%)
  • PASS: status: done_done, add to regression_pool
  • FAIL: log gaps, gates.build = not_started (back to Gate 4), increment attempts
  • 3 failures: status: blocked, escalate to human
  • Run lightweight regression on all done_done screens
Output
state.yaml updated + PROGRESS.md regenerated
Gate 4: Build & Verify Loop
The inner loop that runs for every section of every screen.
4a

BUILD ONE SECTION

Read web source for THIS section only. Extract Tailwind → Dart hex. Note decorative layers. Note ALL click handlers. Write Flutter code. flutter analyze = 0 issues.

4b

DEPLOY & SCREENSHOT

$ ./scripts/migration_verify.sh 00163357C000049 {screen_name}
# build AAR → build APK → install → launch → screenshot
# Output: .migration/verify/{screen}/flutter_iter_{N}.png
⚠
HARD GATE: If script fails, STOP. Fix build errors before proceeding.
4c

VERIFY (Sub-Agent)

Spawn Verifier sub-agent with web screenshot + Flutter screenshot + CSS list + click handlers. Builder CANNOT self-verify.

VISUAL
≥95%
FUNCTIONAL
PASS

PASS
RESPONSIVE
100%
PROD-READY
PASS
4d

ALL PASS → Lock section, move to next. FAIL → Fix gaps, back to 4a (max 5 iterations). 5 FAILs → Ask user for feedback.

Deploy & OTA
Two paths: Shorebird OTA for Dart changes, full APK for native changes.

Shorebird OTA (fast path)

Dart-only changes ship instantly. No app store review. Users get the patch on next app launch.

$ shorebird patch android
# Pushed manually (no CI for now)
When: UI fixes, provider logic, new screens, API changes
Speed: Minutes from merge to user

Full APK Release (slow path)

Native changes (Android/iOS, MethodChannels, Flutter engine). Requires Play Store / App Store review.

$ flutter build aar --no-profile
$ cd android && ./gradlew assembleProductionRelease
When: New MethodChannels, bridge changes, engine upgrade
Speed: Days (app store review cycle)
✔
Hot reload during development: Use dev.sh for Dart-only changes. Check /tmp/dev_hot_reload.state before building. Only full build for first install or native changes.
Parity Scoring: G1-G6
Six dimensions measured. Composite must be ≥ 95% to pass.
GateNameWhat It MeasuresThresholdWeight
G1 Scenario Fidelity Step count + action types match web user flows ≥ 95% 20%
G2 Validation Rules Form validation, input constraints, error messages ≥ 95% 15%
G3 API Fidelity Same endpoints, headers (auth, X-Platform-Id, X-Device-ID), payloads ≥ 95% 25%
G4 State Handling All UI states (loading, empty, populated, error) handled ≥ 95% 20%
G5 Analytics Fidelity Same events fired (Amplitude, CleverTap) with same properties ≥ 90% 10%
G6 Performance Transition <500ms, order <2000ms, scroll 60fps ≥ 80% 10%
Scoring Deductions
-30
Critical: Missing endpoint, broken auth, crash
-15
Major: Wrong header, missing state, wrong navigation
-5
Minor: Color shade off, 1px spacing, animation timing
Composite Formula
composite = G1(0.20) + G2(0.15) + G3(0.25) + G4(0.20) + G5(0.10) + G6(0.10)
Must be ≥ 95% to mark screen as done_done.
