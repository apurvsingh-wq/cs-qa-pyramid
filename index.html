Gates 1-3: Compile + Lint + Static Analysis BLOCKER

The foundation. If code doesn't compile or violates conventions, nothing else runs. These run in ~30 seconds and catch typos, missing imports, type errors, and convention violations before any test executes.

AspectDetails
G1: Compileflutter analyze — zero errors, zero warnings. Catches missing imports, type mismatches, null safety violations.
G2: Lintdart format + custom rules in analysis_options.yaml. Enforces: doc comments on public APIs, Riverpod providers = *Provider, widgets = PascalCase. Pre-commit hook generates missing dartdoc stubs (like Doxygen auto-generation).
G3: StaticCovered by Dart's sound null safety + flutter analyze. No implicit casts, no unused variables/imports. custom_lint for project-specific rules.
Industry refEquivalent to gcc compilation + Doxygen + static cast checks in C++ pipelines.
WhenEvery PR + lefthook pre-push hook (runs locally before code leaves your machine).
Time~30 seconds

Gate 4: Unit + Widget Tests BLOCKER

The workhorse. All existing tests must pass (regression check). New files must have tests. Widget tests are parameterized from Zephyr test cases with expected values from PRO web FORM_SPEC and DATA_MAP.

AspectDetails
Regressionflutter test — all 538+ existing tests must pass. If your new code breaks someone else's test = blocked.
New filesGit diff detects new .dart files in lib/. If no corresponding _test.dart exists, gate warns.
Test sourceZephyr case title (WHAT) + FORM_SPEC/DATA_MAP (expected values from PRO web) = parameterized test. Example: CSK-T988 "Verify stop limit with Min/Max quantity" → testWidgets('min/max quantity — CSK-T988', ...)
Mock dataAPI responses mocked from DATA_MAP (captured from PRO web HARs). Same response shapes, same edge cases.
WhenEvery PR
Time~2-3 minutes

Gate 5: Code Coverage Report REPORT

Coverage report generated and posted as PR comment. Doesn't auto-fail — reviewers enforce thresholds. Traceability matrix auto-attached showing Zephyr coverage.

AspectDetails
Toolflutter test --coverage → lcov report. Per-file % and diff from base branch.
Threshold≥90% for logic files (order_logic, providers). ≥80% for UI widgets (harder-to-test visual paths). Industry standard: ≥95%.
TraceabilityAuto-generated matrix attached to PR: "X of Y Zephyr cases from this screen are covered. Missing: CSK-T989, CSK-T990."
ReviewerIf coverage < threshold, dev must explain why or increase coverage. Reviewer's call to approve or block.
Time~30 seconds (generated during Gate 4 run)

Gate 6: Golden / Visual Regression UI-SPECIFIC BLOCKER

Pixel-diff every widget against approved golden files derived from PRO web screenshots. If Flutter drifts visually from PRO web, the test fails. This is how we guarantee visual parity.

AspectDetails
ToolFlutter's built-in matchesGoldenFile(). Each widget test includes: expectLater(widget, matchesGoldenFile('goldens/stop_limit.png'))
BaselinesPRO web ground truth screenshots → converted to Flutter golden files. Multiple sizes: 320px (iPhone SE), 428px (iPhone 14 Pro Max).
ThresholdFails if any widget drifts >0.5% from golden. Intentional visual changes require updating goldens with --update-goldens.
Industry refAirbnb (Percy), Shopify (Polaris), Google (Flutter Gallery). Chromatic for web, matchesGoldenFile for Flutter.
Time~1-2 minutes (part of flutter test)

Gate 7: Accessibility Audit UI-SPECIFIC WARNING

Every interactive element must have semantic labels, contrast ratios must pass WCAG, tap targets must be ≥48px. Warnings now, blocker by v1.0 launch.

AspectDetails
ToolFlutter's built-in guidelines: meetsGuideline(androidTapTargetGuideline) + meetsGuideline(textContrastGuideline)
ChecksMissing Semantics labels on buttons/inputs, contrast ratio < 4.5:1, tap targets < 48x48dp, missing excludeSemantics on decorative elements.
ComplianceWCAG 2.1 AA. RBI/SEBI may require this for fintech apps. Google Play Store accessibility requirements.
ThresholdWarnings, not blockers (for now). Report attached to PR. Goal: zero warnings by v1.0.
Time~30 seconds (part of widget test suite)

Gate 8: Performance Benchmarks UI-SPECIFIC REPORT

Track app size, frame rate, startup time, memory. Regressions flagged. Trading apps must hit <100ms tick-to-render for live prices.

AspectDetails
Toolflutter test --profile + integration test driver with Timeline tracing. App size: flutter build apk --analyze-size.
MetricsAPK size delta, cold start time, scroll jank (frames >16ms), memory peak during trading screen, widget rebuild count.
ThresholdsAPK +500KB = warning, +2MB = blocker. Startup >3s = warning. Jank >5% = blocker. Tick-to-render >100ms = blocker for trading screens.
Industry refGoogle Flutter perf best practices, Shopify app size budgets.
Time~5 minutes (nightly, alongside E2E)

Gate 9: E2E on Real Device BLOCKER

Full product flows on real Samsung Z Flip6. Build → install → run all YAML smoke flows → screenshot at each step → compare against PRO web ground truth. The final gate.

AspectDetails
ProcessBuild AAR → APK → install on device → authenticate (PIN 1234) → run 49 YAML flows converted to Patrol tests.
VerificationScreenshot captured at each assertion point. Pixel-compared against PRO web ground truth baselines.
ScopeFutures trading, positions, orders, market, options — all critical user paths.
Industry refEquivalent to running on real smart meters in embedded. Same principle: emulators miss hardware quirks, real devices catch them.
WhenNightly. Critical PRs can trigger manually.
Time~15-20 minutes (build + install + 49 flows)
